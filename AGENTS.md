## Encoding Rules

- 所有文件读写一律使用 UTF-8（无 BOM）。
- 禁止使用默认编码、GBK、ANSI。
- 执行任何写文件命令前，必须先检查并确认控制台编码为 UTF-8（`chcp 65001`）。
- 读写文件时必须显式指定 UTF-8 编码。
- 如发现中文乱码（例如“绔炶禌 Service 涓氬姟灞傚鐞?”），不得直接修乱码文本，必须先确定原文（从正确源文件、历史版本或上下文）再替换。
- 若无法确认原文，必须先询问再修改。
- 违反以上规则会再次破坏编码，必须严格遵守。

## Git 提交规范

- 本规范同时适用于 agent 生成提交信息与仓库默认人工提交约定，但本节仅约束提交信息、提交拆分与提交前检查，不扩展到分支命名、PR 模板或合并策略。

### 提交标题格式

- 提交标题固定使用：`<type>: <中文摘要>`
- `type` 只允许使用：`feat`、`fix`、`refactor`、`docs`、`test`、`build`、`ci`、`chore`、`perf`、`revert`
- 冒号必须使用英文半角 `:`
- 摘要必须使用中文，保持单行可读，准确描述本次提交的核心目的
- 摘要末尾不加句号，不使用“更新代码”“修复问题”这类空泛表述
- 本仓库默认不强制 `scope`，如无必要不要自行扩展为 `type(scope): summary`

### 提交拆分规则

- 一次提交只解决一个明确目的，不要把功能、修复、重构、文档混在同一提交中
- 若同次工作包含多类改动，优先按“功能变更 / 配套测试 / 文档更新”拆分提交
- 只有强绑定、无法独立表达的小型配套改动，才允许与主改动放在同一提交
- 不允许把无关格式化、顺手清理、调试残留、临时文件混入业务提交

### 提交前最小检查

- 提交前必须确认暂存内容与提交标题一致，避免标题和实际改动不匹配
- 提交前必须检查 `git diff --staged`，确保没有误提交无关文件
- 若存在可运行的最小验证，应先执行与改动直接相关的测试、构建或静态检查
- 若本次未执行验证，必须在提交说明或最终汇报中明确说明

### 提交信息示例

- `feat: 增加多用户会话隔离`
- `fix: 修复 MCP 请求缺少 user_id 上下文`
- `docs: 重写仓库首页说明`
- `refactor: 拆分语音输入服务工厂`
- `build: 调整 Android APK 打包脚本`
